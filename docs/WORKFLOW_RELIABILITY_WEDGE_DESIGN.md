## Problem Statement [Still Accurate]

OpenClaw is being used for real company workflows, but its behavior feels like a black box in two different ways.

First, individual runs fail and operators cannot quickly tell what happened, where the workflow broke, what changed, or what to fix before the next run.

Second, the agent drifts over time. Important rules live partly in `AGENTS.md`, partly in `MEMORY.md`, partly in chat, and partly in habits the operator never automated. After compaction or repeated edits, the agent can end up following contradictory instructions, forgetting rules that were never persisted, or repeatedly making the same mistake for reasons that are invisible in normal logs.

The immediate pain is not abstract "observability." It is daily business interruption and operator babysitting.

## Demand Evidence [Validated]

- Founder demand is real. The SEO workflow was stealing roughly 30 minutes of CEO time every day.
- Internal failure pattern was sharp: repeated mistakes around the cover image generation step, failure rate above 50% even after repeated self-correction attempts.
- Community pain validated across two threads:
  - Thread 1: runtime-debugging pain (gateway logs, bringing in second debugging agents, custom scripts)
  - Thread 2: control-plane drift pain (contradictory instructions, compaction loss, ad hoc audits)
- Product is live; initial users are onboarding. External demand is being validated in real time.

## Status Quo (Before ClawTrace) [Still the Competitor]

Today the workaround without ClawTrace is:
- babysit the run manually
- inspect gateway logs
- ask OpenClaw to fix itself
- bring in Claude Code / Cursor / Grok to inspect logs and propose changes
- rerun or partially rerun the workflow
- keep manual backups of config
- run context/file audits to see what instructions are loaded
- move important rules from chat into `AGENTS.md` before compaction
- give deterministic work to scripts or cron jobs

## What Shipped in Phase 0

### Core trace views [SHIPPED]

**Execution Path** — collapsible call tree showing every LLM call, tool execution, and sub-agent delegation with per-node cost badges and error highlighting.

**Actor Map** — force-directed diagram of every agent, tool, and model in the run. Immediately understand the shape of a complex multi-agent workflow.

**Step Timeline** — Gantt chart of every span. Find bottlenecks, parallelism, and idle gaps.

**Run Efficiency** — work index (tool density × token efficiency). Banded as: Efficient (61–100), Active but heavy (26–60), High spend low progress (1–25), Idle (0).

### Step detail [SHIPPED]

Full input/output payload per step. Token counts (input, output, cache read, cache write). Cost estimate. Duration. Error message.

### Tracy AI analyst [SHIPPED]

Conversational analysis over live PuppyGraph trajectory data. Ask "why did this run fail?" or "which step cost the most?" and get a specific answer with citations and charts.

### Self-evolve loop [SHIPPED]

`/v1/evolve/ask` endpoint + `clawtrace-self-evolve` skill. Agents can query Tracy about their own trajectories and apply fixes autonomously. Closes the loop: observe → diagnose → recommend → apply → verify.

### Cost attribution [SHIPPED]

Per-span token counts and cost estimates. Visible at trace level and step level.

## What Is Still Phase 1

### State diff and drift detection [PHASE 1]

The original wedge design called for state diff as a Phase 0 feature. This has not yet shipped.

**What this means:** When a user asks "why has my agent gotten weird over time?", ClawTrace cannot yet show a diff of config/AGENTS.md/SOUL.md/skills/plugins between a good run and a bad run.

**Current workaround:** Users can ask Tracy "is my context window growing across sessions?" and Tracy will query the trajectory data for patterns. But a structured state timeline and run-to-run diff view are Phase 1.

See `OPENCLAW_STATE_VERSIONING_RESEARCH.md` for the full design.

### Contradiction detection [PHASE 1]

No automated detection of contradictory instructions across `AGENTS.md` and `MEMORY.md` yet.

**Current workaround:** Users can ask Tracy to analyze recent traces and identify behavioral inconsistencies. A dedicated contradiction scanner is Phase 1.

### Agent Health page [PHASE 1]

Synthesized per-agent status combining error rate, cost trend, and recent Tracy findings. Backend types exist; API endpoint and UI pending.

## Recommended Approach [Still Valid]

Hybrid of A (Workflow Reliability Debugger) and B (Agent Reliability Control Plane).

- **Sell A**: workflow reliability debugger — explain today's failed run, show what changed, distinguish runtime failure from drift.
- **Build toward B**: the full control plane for agent reliability.

The first thing someone buys should be small and urgent: explain today's failed workflow, tell me what changed, and tell me whether I should fix the agent, fix the instructions, or just script this step.

## Phase 1 Wedge Expansion

Building the pieces that complete the original wedge vision:

1. **State Time Machine v0** — state vector capture on run start/end; run-to-run diff view inside trace investigation; correlation between state changes and incident onset.
2. **Contradiction detection** — detect conflicting instructions across `AGENTS.md`, `MEMORY.md`, and persistent instruction files.
3. **Agent Health page** — synthesized status per agent with suggested next fix labeled as: runtime failure / state drift / should be scripted.
4. **Leak detection** — surface avoidable cost drains (background loop overhead, context bloat, model mismatch).
