## Locked Decisions

### Architecture

1. Use existing OpenClaw hooks plus exported telemetry as the runtime seam.
2. Keep a separate operational control path and analytical investigation path.
3. Store operational truth as an append-only control-event log plus materialized current-state views.
4. Make `workflow_run` the execution envelope, with `workflow_contract` and `workflow_trust_state` as workflow-level objects.
5. Define the workflow contract as a machine-first schema with human-readable projection.
6. Use one explicit workflow-run state machine with typed sub-status fields.
7. Keep the hot operational store relational in SaaS with one shared schema (Postgres).
8. Keep Iceberg + PuppyGraph in the MVP for append-only event capture and rich analysis through a cloud-only data-lake sink (no local filesystem persistence mode).
9. Add run-native cost events in the same pipeline:
   - billed cost when the provider emits it
   - deterministic estimated cost fallback when billed cost is missing
10. Require cost attribution pivots in Phase 0:
   - workflow -> trajectory -> step class -> model/tool
   - cost-per-success and retry-loop cost as first-class derived metrics

### Code Quality

1. Build one shared decision kernel with explicit modes: `preflight`, `pre_action`, `resume`, `canary`.
2. Organize Phase 0 by feature-centered modules, not a service-per-capability graph.
3. Keep all control artifacts deterministic; use model assistance only for wording and presentation.

### Tests

1. Build a deterministic OpenClaw hook replay harness.
2. Create a golden fixture suite for contract bootstrap and drift cases.
3. Add invariant/property-style tests for state-machine legality, auth semantics, and resume rules.

### Performance

1. Materialize one hot control snapshot per workflow/run for the decision kernel.
2. Return decisions first and run memo / explanation / graph enrichment asynchronously.
3. Cost aggregation must not block control decisions; heavy rollups run asynchronously over cached projections.
4. Cost precision class (`billed` vs `estimated`) must be explicit in all APIs and UI payloads.

## Cost Audit and Cost Control Vertical Slice (Added 2026-03-24)

Phase 1 engineering deliverable should include one end-to-end cost control loop, not only passive attribution.

### Required detectors

1. `background_overhead_detector`
- detect spend from periodic/background checks
- label with trigger source and confidence

2. `context_bloat_detector`
- detect oversized always-loaded context payloads
- estimate avoidable token overhead

3. `model_mismatch_detector`
- detect low-complexity tasks routed to premium model tier
- estimate cheaper-route alternative

4. `retry_loop_detector`
- detect repeated failing loops with rising spend
- mark as high-risk for budget guardrails

### Required control actions

1. background routing policy recommendation
2. context loading policy recommendation
3. model routing policy recommendation
4. budget/rate guardrail recommendation
5. stable-context cacheability recommendation

All actions remain recommendation-first with explicit operator confirmation in Phase 1.

### Required verification outputs

After a control action, verification must show:
1. cost-per-success delta
2. avoidable-spend ratio delta
3. trust-state or quality regression signal (if any)
4. confidence/precision class (`billed` or `estimated`)

## Failure Modes

| Codepath | Realistic production failure | Test planned | Error handling planned | User-visible outcome |
|---|---|---:|---:|---|
| Run start contract binding | Latest contract changed concurrently and wrong version is attached | Yes | Yes | Clear stale-state / retry message |
| Preflight | Missing prerequisite or deterministic verifier failure | Yes | Yes | Clear block with next action |
| Pre-action authorization | Allow token reused or issued for stale state | Yes | Yes | Clear deny / defer |
| Unknown safety timeout | Control loop cannot establish safety in time | Yes | Yes | `Awaiting Confirmation` |
| Mid-run state change | Hot policy/config changes after run start | Yes | Yes | Clear revalidation requirement |
| Resume flow | Deferred action resumes without fresh validation | Yes | Yes | Clear denial or renewed defer |
| Verification | Target side effect completed but one verifier is unknown | Yes | Yes | `Partially Verified` with breakdown |
| Canary | Shadow target unavailable or misrouted | Yes | Yes | Read-only fallback or explicit canary failure |
| Side-effect journal | Duplicate commit/deploy/send attempt | Yes | Yes | Clear duplicate prevention message |
| Operational projections | Current-state projection lags event log | Yes | Yes | Clear retry / stale-state handling |
| Iceberg append | Analytical sink is slow or temporarily unavailable | Yes | Yes | Control decision succeeds; analytics replay later |
| Incident memo | Memo drifts from structured evidence | Yes | Yes | Deterministic facts with model wording only |
| Cost attribution | Provider billed cost is absent for some runs | Yes | Yes | Show explicit `estimated` label with deterministic fallback |
