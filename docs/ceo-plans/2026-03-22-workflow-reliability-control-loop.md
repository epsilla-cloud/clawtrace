## Vision

### 10x Check
ClawTrace becomes the workflow reliability control plane for AI agents: before a critical workflow runs, it knows the intended contract, checks whether state drift or missing dependencies make the run risky, verifies whether the side effect actually happened, classifies failures as runtime vs state vs should-be-scripted, and converts every incident into a reusable alert, memo, and regression asset.

## Core Architectural Decisions

### Product shape
- ClawTrace should launch as a **workflow reliability control loop** for one critical workflow, not as a broad generic observability suite.
- The long-term moat is **workflow contract + control authority**, not just data exhaust or chat UX.

### Workflow contract
- The workflow contract is the primary control object.
- ClawTrace bootstraps the contract from traces, state, and observed behavior, then the user explicitly confirms or edits it.
- The confirmed contract becomes the durable source of truth.
- The contract is a versioned explicit artifact with a narrow structured schema and limited escape hatches.

### Runtime control model
- ClawTrace should observe all relevant runtime spans/events, but only intervene at explicit control points.
- Risky mutations require synchronous pre-action hooks.
- Runtime behavior maps into contract-defined logical steps; ClawTrace does not rely on model self-narration for step identity.
- Runs bind to a contract version at start for attribution, but mid-run upstream live updates mark the run as `State Changed During Run` and require revalidation at the next risky boundary.

### Deployment modes
- ClawTrace should ship SaaS at launch while preserving local-first trust and testing.
- Control decisions execute locally first; cloud is optional for collaboration, retention, and enterprise operations.
- Local and SaaS share one semantic/control core.
- The intended infrastructure difference is storage/compute:
  - local: local filesystem plus embedded/open graph stack
  - cloud: Iceberg plus PuppyGraph
- Above storage, one canonical semantic layer should serve both modes.

### Verification and trust
- Outcome verification is hard-check first, soft-eval second.
- Verification architecture should be a small adapter interface plus one concrete verifier pack for the initial workflow.
- Trust should be represented as an explicit finite-state model, not a standalone numeric score in Phase 0.
- Partial verification is first-class and should show a verifier breakdown (`success`, `fail`, `unknown`) in addition to the headline state.

## Error and Rescue Policy

- Deterministic preflight/canary failures block risky mutations.
- Heuristic risk signals warn and degrade trust, but do not directly block in Phase 0.
- If ClawTrace cannot establish safety, the workflow enters `Awaiting Confirmation` rather than silently failing open or closed.
- Deferred actions preserve intended action context and idempotency keys, but must be revalidated before resume.
- Each incident should recommend one primary next action plus secondary options.
- Recovery recommendations should come from guardrailed dynamic selection within a bounded playbook, not from fully unconstrained agent choice.
- Mutating steps require side-effect journaling and idempotency keys.

## Security Decisions

- ClawTrace is an explicit authorization authority for gated mutations.
- Allow decisions are single-use, context-bound, and short-lived.
- Raw payload access is allowed by default for debugging power, but opt-out and redaction must be easy and explicit.
- Shared artifacts default to redacted/summarized output.
- RBAC and ABAC should define who may share, who may elevate, and how redaction is performed.
- Prompt/trace injection defenses require a two-stage boundary:
  - untrusted payloads -> structured evidence extraction
  - structured evidence -> control decisions / memos / recommendations

## Data Flow and Interaction Decisions

- Control objects use optimistic concurrency with explicit stale-state rejection.
- Shadow-target canaries are preferred; otherwise canaries fall back to read-only verification.
- The Determinism Advisor can influence trust state and alerts, but cannot directly block in the MVP.

## Test and Rollout Strategy

- Phase 0 testing is anchored on a workflow scenario matrix for one critical workflow.
- The test pyramid is scenario-driven and layered:
  - scenario matrix
  - unit tests for decision logic
  - integration tests for runtime/storage hooks
  - a small number of end-to-end control-loop tests
- Trust/defer/authorize/resume behavior must have explicit state-machine tests.
- Canary behavior needs a dedicated safety suite.
- Rollout progression should be:
  - observe
  - recommend
  - gate
- Feature rollout should be per workflow and per capability.

## UX Decisions

- The primary MVP surface is a workflow cockpit, not a blank chat home.
- Chat should be the guided onboarding and investigation companion.
- One-time chat onboarding should bootstrap durable artifacts:
  - workflow contract
  - risky-step map
  - verifier pack
  - preflight settings
  - initial trust state

### Rollout progression
```text
OBSERVE  -->  RECOMMEND  -->  GATE
  |            |              |
  |            |              +-- allow/deny/defer on risky mutations
  |            +-- trust state, memos, warnings, next action
  +-- traces, verification, state diff, side-effect journal
```

### Decision ladder
```text
deterministic fail   -> BLOCK
heuristic risk       -> WARN
unknown safety       -> AWAITING_CONFIRMATION
partial verification -> PARTIALLY_VERIFIED + verifier breakdown
```
